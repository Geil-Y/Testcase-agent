# Clarification-First Review Pipeline Implementation Issues

## Context

Build a replacement generation pipeline for Testcase-agent.

The new pipeline is clarification-first, confidence-routed, human-reviewable,
and review-memory-assisted. It replaces the old generation flow instead of
remaining compatible with it.

New main entry point:

```text
python -m review_pipeline.cli
```

Core flow:

```text
Input requirements
  -> LLM-A Requirement Decomposer
  -> clarification_review.json + clarification_review.html
  -> human edits clarification_review.json
  -> validate-review
  -> clarified test basis
  -> LLM-B Case Intent Planner
  -> case_intent_review.json + case_intent_review.html
  -> human edits case_intent_review.json
  -> validate-review
  -> approved_case_plan.json
  -> LLM-C Case Writer
  -> generated_cases.json
  -> existing evaluator/report
  -> optional import-memory
```

Out of scope for phase 1:

- Main UI integration.
- Excel import.
- Automatic human skipping.
- Automatic prompt optimization.
- Embedding search.
- Code-driven automatic approve/reject.
- Full replacement of existing evaluator logic.
- `self_check` / repair stage.

Design constraints:

- Code remains plumbing only.
- Prompt owns generation philosophy and semantic judgment.
- JSON is the source of truth.
- HTML is only the human review view.
- LLM stage outputs are parsed into JSON artifacts. Review HTML is generated by code from JSON, not authored by the LLM.
- Human edits JSON directly in phase 1.
- `validate-review` protects schema and required decision fields.
- Review Memory is written only by explicit `import-memory`.
- Review Memory is advisory, not authoritative. It may affect confidence, routing, and reviewer hints, but must not introduce expected behavior, thresholds, timing, signals, recovery behavior, diagnostics, or case intents not supported by the current requirement/test basis.
- `pattern_tags` are deterministic memory indexes derived by code, never by LLM or direct human editing.

## Issue 2: Implement Confidence Aggregation And Color Routing

Type: AFK

Blocked by: Issue 1.

### What To Build

Implement confidence aggregation for clarification items and case intent items. LLMs provide driver scores and reasons; code only aggregates, maps to routing color, and applies bounded historical adjustment.

Clarification confidence drivers:

```text
trigger_clarity
expected_behavior_clarity
known_info_sufficiency
ambiguity_resolution
historical_pattern_support
```

Case intent confidence drivers:

```text
requirement_basis_strength
separate_case_value
missing_info_handling
historical_decision_support
```

Routing thresholds:

```text
green  >= 0.85
blue   >= 0.65 and < 0.85
orange >= 0.40 and < 0.65
red    < 0.40
```

Historical adjustment is bounded to `-0.10` through `+0.10`.

### Acceptance Criteria

- [ ] Confidence aggregation is deterministic.
- [ ] Driver scores must be between `0.0` and `1.0`.
- [ ] Missing historical driver defaults to `0.5`.
- [ ] Routing colors match the agreed thresholds.
- [ ] Historical adjustment cannot exceed `-0.10` or `+0.10`.
- [ ] Clarification labels:
  - green: `Clear`
  - blue: `Minor ambiguity`
  - orange: `Review required`
  - red: `Clarification required`
- [ ] Case intent labels:
  - green: `Strong intent`
  - blue: `Review recommended`
  - orange: `Review required`
  - red: `Do not generate yet`

### Testing Requirements

- [ ] Unit tests for exact threshold boundaries.
- [ ] Unit tests for invalid driver values.
- [ ] Unit tests for average confidence with no history.
- [ ] Unit tests proving history adjustment is bounded.
- [ ] Unit tests for clarification and case intent label mapping.

## Issue 4: Implement Deterministic Pattern Tag Subsystem

Type: AFK

Blocked by: Issue 1.

### What To Build

Implement deterministic `pattern_tags` as evidence-backed memory indexes.

`pattern_tags` must never be generated by the LLM and must not be directly edited by humans. They are derived from structured fields, reason codes, coverage dimensions, missing information categories, ambiguity types, and conservative text detectors.

Add:

```text
review_pipeline/pattern_tags.yml
review_pipeline/pattern_tag_rules.py
```

Each derived tag must include:

```text
tag
tag_strength       # confirmed | candidate
source             # reason_code | ambiguity_type | missing_category | coverage_dimension | text_detector
rule_id
evidence_text
confidence
```

### Acceptance Criteria

- [ ] Tag registry defines allowed tags, labels, descriptions, groups, and allowed sources.
- [ ] Unknown tags are rejected.
- [ ] Reason codes derive confirmed tags.
- [ ] Ambiguity types derive confirmed tags.
- [ ] Missing info categories derive confirmed tags.
- [ ] Coverage dimensions derive confirmed tags.
- [ ] Text detectors derive candidate tags only.
- [ ] Candidate tags never act as authority.
- [ ] Tag derivation does not approve, reject, create, or delete case intents.

### Testing Requirements

- [ ] Unit tests for reason-code tag derivation.
- [ ] Unit tests for ambiguity-type tag derivation.
- [ ] Unit tests for missing-category tag derivation.
- [ ] Unit tests for coverage-dimension tag derivation.
- [ ] Unit tests for conservative text detectors:
  - response-time bound candidate
  - timing maturity candidate
  - diagnostic clear candidate
  - persistence candidate
  - logging/record candidate
- [ ] Unit test that LLM-provided arbitrary tags are ignored or rejected.

Implement retrieval from Review Memory so the system can use accumulated human
review experience as advisory support for future runs.

scoring. Review Memory must remain advisory: it may influence confidence,
routing, and reviewer hints, but it must not create, approve, reject, merge,
split, or delete case intents by itself.

Retrieval should support:

- same `source_requirement_hash` lookup
- same `test_basis_hash` lookup
- similar item lookup by overlapping deterministic `pattern_tags`
- similar item lookup by overlapping reason codes
- decision statistics for matching tags/reasons

Historical confidence adjustment remains bounded to `-0.10` through `+0.10`.

### Acceptance Criteria

- [ ] Retrieval APIs can query prior clarification memory items.
- [ ] Retrieval APIs can query prior case intent memory items.
- [ ] Retrieval APIs return evidence for why a historical item matched.
- [ ] Same requirement/test-basis matches can produce stronger adjustment than broad tag-only matches.
- [ ] Pattern-tag matches are advisory and never authoritative.
- [ ] Historical support can populate `historical_pattern_support` for clarification confidence.
- [ ] Historical support can populate `historical_decision_support` for case intent confidence.
- [ ] Historical support is surfaced in review JSON and HTML as reviewer hints.
- [ ] Historical adjustment cannot exceed the configured bound.
- [ ] If no relevant memory exists, historical support remains neutral.

### Testing Requirements

- [ ] Unit tests for same requirement hash retrieval.
- [ ] Unit tests for same test basis hash retrieval.
- [ ] Unit tests for tag-overlap retrieval.
- [ ] Unit tests for reason-code-overlap retrieval.
- [ ] Unit tests proving retrieval does not mutate review decisions.
- [ ] Unit tests for bounded positive and negative historical adjustment.
- [ ] Integration test: import a reviewed item, run a later similar item, and verify historical support appears in output.

For each test case you plan to generate, produce one case_intent element with a coverage attribute. Valid coverage values: normal_behavior, boundary_or_threshold, fault_or_protection, state_transition, observability. Case count is determined by verification value, not by coverage labels. Before creating any , apply this gate: Create a new case only if changing the tested condition changes at least one of: - the requirement-relevant expected behavior, - the pass/fail judgment, - the evidence needed to prove the requirement. Do not create a separate case only because: - the wording differs, - the coverage label differs, - the observation method differs, - the case restates the same trigger and the same expected behavior, - the case checks that an already expected result is observable. Different observable evidence paths justify separate cases only when they correspond to different requirement-defined behaviors or acceptance criteria. If a signal/state assignment is the observable evidence for the same required response, keep it in the same case as that response. Do not create one case for the response timing and another case for the resulting state update unless the selected requirement gives separate acceptance criteria for them. Multi-condition triggers: - Do not enumerate OR/AND combinations by default. - Do not create one case for every satisfied or unsatisfied condition combination. - Create a separate case only when that condition change changes expected behavior, pass/fail judgment, or required evidence. - For multi-condition negative cases, change one requirement-critical trigger condition to not satisfied and keep the other trigger conditions satisfied. - Do not add combined-negative cases unless the selected requirement explicitly defines different behavior for the combined condition. Threshold boundary values: - The "triggers" condition and the "does not trigger" condition are separate cases only when they produce different expected behavior. - Use semantic relations such as above threshold, below threshold, at threshold, shorter than timing, or longer than timing. - Do not invent numeric epsilon values. - A boundary case is not justified if it repeats the same trigger and same expected behavior as another case. Debounce / confirmation / hold-time conditions: - "duration mature" and "duration not mature" are separate cases. - If the selected requirement explicitly describes debounce reset, intermittent fault, timer restart, or recovery before maturity, create separate cases for those behaviors. - Do not create these cases when the selected requirement does not define those behaviors. Response-time / within-time-bound requirements: - A response-time bound defines an acceptance deadline for the required response. - Create positive verification cases that prove the required response occurs within the bound. - Do not create late-response, missed-deadline, no-response, or fault-state cases unless the selected requirement explicitly defines expected behavior for those conditions. - Do not infer diagnostic behavior, fault state, retry behavior, recovery behavior, or timeout handling from a missed response-time deadline. - A response-time bound is not debounce, confirmation, hold-time, or timing maturity. - Do not create a "duration not mature -> no trigger" case for response-time-bound requirements. - Never create a case whose expected behavior is about what happens after a requirement deadline is missed unless the selected requirement explicitly states that behavior. - For chained response-time requirements, create at most one case per distinct response-time acceptance criterion. - Include the associated state/signal update as expected evidence in that same case. - Do not add extra cases for restating the same state transition. Explicit requirement-defined behaviors stay separate: - diagnostic clear, - latch persistence, - power-cycle persistence, - timeout, - reset, - recovery, - mode transition, - different DTC/CAN/log/record acceptance criteria. Do not create case intents that introduce new numeric test points, recovery scenarios, clear/reset actions, timeout behavior, or state transitions unless they are explicitly stated in the selected requirement. A case_intent must not contain any numeric value unless that exact value appears in the selected requirement. Do not derive or choose substitute numeric values.

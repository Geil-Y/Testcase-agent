# Review Pipeline Artifacts

## JSON Source-of-Truth Rule

JSON is the source of truth. HTML is only the human review view.
LLM stage outputs are parsed into JSON artifacts. Review HTML is generated by
code from JSON, not authored by the LLM.

## Artifact Catalog

### `requirements.json` (input)
Produced by: External (Excel import or hand-authored).
Consumed by: `prepare-clarification-review`.

```json
[{
  "requirement_key": "BMS_REQ_001",
  "description": "...",
  "function_name": "...",
  "supplementary_info": "..."
}]
```

### `clarification_review.json`
Produced by: `prepare-clarification-review` (LLM-A).
Consumed by: Human reviewer.

Decomposes a requirement into facts, ambiguities, clarification questions,
and safe generation policy. Each ambiguity item has confidence drivers,
recommended review decision, and routing.

### `clarified_test_basis.json`
Produced by: `validate-review` (after human edits).
Consumed by: `prepare-intent-review` (LLM-B).

Resolved requirement understanding with approved facts and resolved
ambiguities. If any item is blocked, planning is blocked.

### `case_intent_review.json`
Produced by: `prepare-intent-review` (LLM-B).
Consumed by: Human reviewer.

Proposed case intents with coverage dimensions, confidence drivers,
routing colors, and recommended review decisions.

### `approved_case_plan.json`
Produced by: `validate-review` (after human edits).
Consumed by: `generate-cases` (LLM-C).

Final approved intents. Contains only case-writer-ready intents.
Preserves traceability to original intent ids and review decisions.

### `generated_cases.json`
Produced by: `generate-cases` (LLM-C).
Consumed by: `evaluate`, existing evaluator/report utilities.

One test case per approved intent. Each case has full test structure
(title, objective, pre_condition, steps, post_condition) plus
traceability fields.

### `evaluation_results.json` / `evaluation_summary.json`
Produced by: `evaluate`.
Consumed by: Reporting.

Hard-rule evaluation results and pass/fail summary.

## Review Decisions

### Clarification decisions
`approve` | `clarify` | `mark_needs_review` | `block` | `edit`

### Case intent decisions
`approve` | `reject` | `revise` | `merge` | `split` | `defer`

Non-approve decisions require reason codes. Some require reason text.
See `src/testcase_agent/review_pipeline/reason_codes.yml` for the full registry.

## Confidence Routing

| Color | Range | Clarification label | Case intent label |
|---|---|---|---|
| Green | >= 0.85 | Clear | Strong intent |
| Blue | >= 0.65, < 0.85 | Minor ambiguity | Review recommended |
| Orange | >= 0.40, < 0.65 | Review required | Review required |
| Red | < 0.40 | Clarification required | Do not generate yet |

## Pattern Tags

`pattern_tags` are deterministic memory indexes derived by code from:
- Reason codes (confirmed)
- Ambiguity types (confirmed)
- Missing info categories (confirmed)
- Coverage dimensions (confirmed)
- Text detectors (candidate only)

Tags are never generated by LLM and never directly edited by humans.
Candidate tags never act as authority.

## Review Memory

Stored in SQLite. Tables: `review_sessions`, `clarification_memory_items`,
`case_intent_memory_items`, `memory_item_tags`.

Review Memory is advisory, not authoritative. It may influence confidence
and routing, but must not create, approve, reject, or modify case intents.
